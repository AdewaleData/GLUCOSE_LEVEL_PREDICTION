Progress Update Report — Non-Invasive Glucose Estimation (ML Prototype)

Project: non-invasive glucose monitoring using physiological sensing and machine learning. Reporting period: current milestone. Audience: client and stakeholders.

Regarding figures: you do not have to add charts to this narrative report for it to be acceptable. A concise written progress update is often enough, especially when the analysis notebook already contains the visuals and the client can open it if needed. Optional additions would be one figure on pipeline scale (rows retained through cleaning) and one on target distribution in clean-dataset-relevant.csv, only if the client prefers a single glance at evidence; otherwise prose and the listed deliverables are sufficient.

This update summarizes where the project stands. The learning problem is framed as regression: predicting continuous Glucose_level in mg/dL from PPG-derived features, heart rate, and demographics. The intent remains a non-invasive workflow suitable for an engineering prototype, with transparent steps that can later align to hardware such as an ESP32-based PPG front end, without claiming medical certification.

Work completed to date covers data auditing and integrity. The full raw file clean-dataset.csv (on the order of eight hundred forty-five thousand rows) was ingested and profiled. Several columns behaved as metadata or ordering signals and would cause data leakage if fed to a model, inflating scores without true physiological learning; this risk was documented and those fields are excluded from the modeling table. Large amounts of duplication and repeated physiological patterns were also characterized so that training does not memorize redundant rows or contradict itself. The task was confirmed as regression on Glucose_level rather than classification.

A reproducible script build_relevant_dataset.py now produces clean-dataset-relevant.csv. That dataset drops leakage-prone fields, removes duplicates in line with the pipeline, applies plausible human ranges on vitals and demographics, and applies robust filtering on signal-scale peak columns understood as waveform amplitudes rather than calibrated blood pressure in millimetres of mercury. Where the same physiological state appeared with multiple glucose labels, targets were merged to one value per state using the median so the model is not trained on inconsistent input-output pairs. Interpretable engineered features were added, such as pulse-amplitude style composites, area normalized by heart rate and pulse swing, and body size metrics including BMI and BSA-style surfaces, while avoiding redundant interaction spam. A companion file clean-dataset-relevant-report.json records row counts, steps applied, and final column names. After cleaning, the modeling set is on the order of two thousand rows spanning roughly two dozen participants, which is adequate to validate the end-to-end pipeline and first models, with the understanding that broad clinical generalisation requires more diverse data.

Exploratory and domain analysis is captured in 01_data_domain_analysis.ipynb, which walks step by step from the original CSV to the relevant CSV, including distributions, a demonstration of why patient-wise or group-based splitting matters for honest evaluation, checks that each unique physiological state maps to a single glucose target, and discussion of how standard clinical-style glucose bands apply to this file. Most measured values in the cleaned target fall in a band that standard thresholds label as normal because the current sample does not deeply cover hypoglycaemia or severe hyperglycaemia; that is a limitation of coverage, not an error in the analysis plots.

Key findings from the data work are as follows. Leakage control and subject-aware splitting are essential; random row splits can massively overstate performance when metadata or repeated structure leaks identity-like information. The prototype dataset’s glucose range is narrow compared with full clinical reality, so any band-based dashboard or “high versus low” narrative must be qualified; primary modelling should stay on continuous glucose prediction. The pipeline is scripted and the notebook is shareable, which supports reproducibility for supervision and handover.

Current status at this milestone is that data ingestion, cleaning, first-generation feature engineering, exploratory analysis, and written documentation for the data path are complete for the prototype scope. Model training with rigorously grouped validation, formal error metrics such as mean absolute error and root mean square error, and integration toward the hardware path are the next phase and are not yet delivered.

Limitations stated explicitly for stakeholders include limited sample diversity and label range relative to a deployable medical product, absence of certification, and the inherent difficulty of inferring blood glucose from indirect signals; any future accuracy claims will depend on validation design, especially evaluation that holds out whole participants rather than shuffling rows.

Recommended next steps are to train baseline regressors such as random forest or gradient boosting on clean-dataset-relevant.csv with scaled inputs and grouped cross-validation by patient identifier, to report MAE and RMSE with patient-wise holdout, to add interpretability summaries such as feature importance aligned with the engineered physiological features, to keep training metrics clearly separated from any user-facing health claims in a demo interface, and to plan future data collection with time-aligned reference glucose and ideally raw PPG segments from the intended sensor chain.

Deliverables available now besides this report include clean-dataset.csv as the original input, clean-dataset-relevant.csv as the cleaned and engineered training table, clean-dataset-relevant-report.json as the automated build summary, build_relevant_dataset.py as the reproducible pipeline, 01_data_domain_analysis.ipynb as the narrative analysis workbook, and _gen_notebook.py as an optional utility to regenerate the notebook from source if the team maintains it that way.

This milestone establishes the data and methodology foundation for the machine learning prototype. The following phase should focus on model fitting, honest evaluation, and a cautious line between technical results and any messaging about health impact.

This report describes technical progress for a prototype engineering effort and does not constitute medical advice, clinical validation, or regulatory documentation.
